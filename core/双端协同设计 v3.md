# Hermes × WorkBuddy 双端协同设计 v3.0

> 2026-05-03 | 基于v2.0升级
> 核心变化：**Hermes（VPS 24h）为唯一主中枢，WorkBuddy（Mac）为白天高频协作前台**

---

## 零、先说本质

你的真实痛点不是"双端同步"，而是：

> **白天在Mac上用WorkBuddy聊了1小时、做了关键决策、产出了业务方案——晚上跟Hermes对话时，Hermes什么都不知道。**

所以设计的核心不是"怎么同步文件"，而是：

> **怎么让WorkBuddy产生的高价值上下文实时被Hermes掌握，确保任何时候切换到Hermes都能无缝接续。**

## 二、上下文同步机制（核心）

### 2.1 WorkBuddy → Hermes（白天实时掌握）

**问题**：锋哥白天在Mac上跟WorkBuddy聊了很多，Hermes不知道。

**解法**：三层捕获

| 层级 | 机制 | 说明 |
|------|------|------|
| **实时层** | WorkBuddy对话日志 | 每次对话自动写入 `daily/YYYY-MM-DD-workbuddy.md` |
| **提取层** | 信号捕获 | 每次对话结束提取：关键决策/业务数据/待办/新增实体 |
| **收割层** | Hermes定时收割 | Hermes cron每天21:00读取WorkBuddy日志→提炼→写Hermes记忆 |

**信号捕获标准**（每次对话结束必须提取）：
1. 关键决策（做了什么决定）
2. 业务数据（数字变化、人名、时间线）
3. 待办承诺（说了要做什么）
4. 新增实体（首次出现的人名/概念/事件/产品）

### 2.2 Hermes → WorkBuddy（夜间补充）

- Hermes每日收割后生成摘要
- WorkBuddy下次启动时pull Git即可获取最新状态

### 2.3 信息流

```
WorkBuddy对话 → daily/YYYY-MM-DD-workbuddy.md → Git push
                                                    ↓
Hermes cron 21:00 ←──────────────────────────── Git pull
        ↓
  提炼关键信息 → 写入memos记忆 → 发飞书简报
```

## 四、交班协议

### 4.1 WorkBuddy → Hermes（白天→晚上）

**触发条件**（任一）：
1. 锋哥说"走了"/"晚安"/离开Mac
2. 22:00 Hermes自动收割
3. Hermes主动提醒

**交班产出**：Hermes读取WorkBuddy日志→提炼→写入memos→发飞书简报

### 4.2 Hermes → WorkBuddy（夜间→白天）

- WorkBuddy下次启动时 `git pull` 获取最新
- 核心状态：Hermes记忆是主记忆源，WorkBuddy MEMORY.md是辅记忆

## 五、容灾

| 故障场景 | 影响 | 应对 |
|---------|------|------|
| WorkBuddy挂 | 白天不方便，但Hermes 24h可用 | 飞书直连Hermes |
| Hermes挂 | 没有定时、没有飞书通道 | WorkBuddy独立运作，日志继续写 |
| 双挂 | 全部中断 | 文件系统在，恢复后自动同步 |
| Git仓库挂 | 同步中断 | 两端各自继续，恢复后rebase |

## 七、总纲

Hermes是唯一主中枢（24h在线、记忆主存储、飞书通道、定时任务），WorkBuddy是白天高频协作前台（Mac本地执行、core文件修改权）。两者通过Git共享仓库同步——WorkBuddy每次对话写日志，Hermes每天收割提炼。任何时候切换到Hermes都能掌握全部上下文。
