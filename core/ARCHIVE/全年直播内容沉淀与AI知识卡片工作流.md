# 全年直播内容沉淀与 AI 知识卡片工作流

<aside>
🎯

这份文档用于把「全年直播录制 → 转写 → 洗稿沉淀 → 年度经营节点归档 → AI 知识卡片 → 培训 / SOP / 人机协作」设计成一套可持续运行的知识生产系统。

</aside>

## 1. 我对你需求的理解

你现在要解决的不是单篇直播整理问题，而是要建立一套面向托管机构经营的 **全年直播知识沉淀系统**。

你的真实需求可以拆成四层：

### 1.1 表层需求：直播内容自动沉淀

你希望每场直播结束后，系统能自动完成：

1. 自动录制直播视频
2. 视频入库
3. 视频转音频 / 文案
4. 转写稿清洗
5. 去掉直播话术、闲聊、重复内容、销售话术
6. 保留真正有用的方法论、流程、案例、表格、话术
7. 生成结构化文档
8. 方便未来 AI 快速引用

### 1.2 中层需求：按年度经营节点组织

因为这些直播不是随机内容，而是围绕托管机构全年经营节奏展开：

- 寒假招生
- 春季开学
- 晚辅稳定
- 期中复盘
- 暑假招生
- 暑假托管
- 秋季招生
- 团队建设
- 年度复盘
- 春季续费

所以你希望所有直播不只是“整理成文档”，而是能自动归入一张 **年度运营行事历**。

### 1.3 深层需求：形成 AI 可调用知识资产

你希望这些内容未来能被 OpenClaw / Hermes / 其他 Agent 使用，而不是只给人看。

这意味着文档必须具备：

- AI 速读卡
- 适用场景
- 可回答的问题
- 不适用边界
- 核心结论
- SOP
- 检查表
- 话术模板
- 知识卡片
- 关联经营节点

也就是说，你要的不是“文章库”，而是 **AI 知识弹药库**。

### 1.4 最终目标：人和 AI 协作的经营知识操作系统

最终你想要的是：

> 把全年直播内容转化为一套可培训员工、可辅助校长决策、可让 AI 引用、可沉淀组织能力的年度经营知识系统。
> 

它服务于三类对象：

| 对象 | 用途 |
| --- | --- |
| 你自己 / 校长 | 做年度规划、月度复盘、经营决策 |
| 团队老师 | 做培训、SOP、检查表、话术执行 |
| AI Agent | 快速检索、引用、生成方案、辅助复盘 |

## 2. 核心判断：不要做“洗稿系统”，要做“知识对象生产线”

如果只是把直播稿整理得通顺，本质上仍然是文章整理。

但你的目标更高级：你要把每场直播处理成一个可以被系统调用的 **知识对象**。

一场直播最终应该产出四类资产：

| 资产类型 | 用途 |
| --- | --- |
| 原始素材 | 视频、音频、原始转写稿，保留证据链 |
| 结构化文档 | 给人阅读、培训、复盘 |
| AI 知识卡片 | 给 Agent 快速引用 |
| SOP / 表格 / 话术 | 给业务直接执行 |

所以正确的系统不是：

```
视频 → 文案 → 洗稿 → 文档
```

而应该是：

```
视频 → 转写 → 清洗 → 主题切块 → 知识结构化 → 年度节点归档 → 知识卡片拆解 → SOP / 话术 / 培训材料
```

## 3. 总体系统设计

### 3.1 系统名称

**全年直播经营知识沉淀系统**

也可以命名为：

- 直播知识飞轮
- 托管经营年度知识库
- AI-OS 内容沉淀流水线
- OpenClaw 直播知识加工线
- Hermes 年度经营知识归档器

### 3.2 总流程

```
全年直播录制
	↓
视频自动保存
	↓
音频提取 / 语音转写
	↓
原始转写稿入库
	↓
Agent 1：基础清洗
	↓
Agent 2：去噪与主题切块
	↓
Agent 3：结构化沉淀
	↓
Agent 4：年度经营节点归档
	↓
Agent 5：知识卡片拆解
	↓
Agent 6：SOP / 表格 / 话术生成
	↓
人工抽检与评级
	↓
进入正式年度经营知识库
```

### 3.3 运行原则

1. **先搭骨架，再自动化**
    - 先建立年度行事历和知识分类法，再让 Agent 自动归档。
2. **先做样稿，再批量处理**
    - 先用 1-3 篇直播打磨标准，再批量跑全年内容。
3. **先知识卡片化，再让 AI 大规模引用**
    - 长文适合人看，知识卡片适合 Agent 调用。
4. **全自动生成，半自动入库**
    - Agent 可以自动处理，但正式进入培训 / SOP 的内容需要人工抽检。
5. **保真优先于漂亮**
    - 宁愿保留“待复核”，也不要让 Agent 编造不存在的观点。

## 9. 示例知识卡片

## 9.1 知识卡片：寒假托管定价公式

### 基本信息

- 所属系统：招生增长系统 / 产品项目系统
- 所属年度节点：11月寒假招生预热、1月寒假托管执行
- 所属主题：寒假托管
- 适用对象：校长、招生负责人
- 置信度：中高，需结合本地消费水平调整

### 解决的问题

解决托管机构不知道寒假班如何定价、定价过低影响定位、定价过高担心家长不接受的问题。

### 核心结论

寒假托管服务时长长于学期托管，因此日单价应高于学期托管日单价，一般可按学期托管日价的 130%-150% 设计。

### 操作步骤

1. 计算学期托管单日价格。
2. 如果是全托机构，可乘以 130% 左右。
3. 如果是纯晚辅机构，可乘以 150% 左右。
4. 得到寒假托管日价。
5. 根据本地消费水平和竞争情况微调。
6. 先建立原价，再通过折扣、团购、学费共享等方式做活动。

### 判断标准

- 家长能理解寒假班服务时间更长、内容更多。
- 原价能支撑机构定位。
- 活动价不破坏长期价格体系。
- 招生方案能兼顾现金流和转化。

### 常见错误

- 直接低价引流，破坏品牌定位。
- 只看同行价格，不计算自身成本。
- 没有区分原价和活动价。
- 没有把寒假班和收心班区分开。

### AI 调用说明

当用户问以下问题时可引用：

- 寒假托管怎么定价？
- 寒假班收费多少合适？
- 新校区寒假班能不能低价引流？
- 寒假班原价和活动价怎么设计？

### 使用边界

价格类内容必须结合当地消费水平、机构定位、成本结构、师资配置人工确认。

## 9.2 知识卡片：作业规划表

### 基本信息

- 所属系统：托管交付系统
- 所属年度节点：1月寒假执行、7月暑假执行、2月收心班
- 所属主题：假期作业管理
- 适用对象：校长、班主任、托管老师
- 置信度：高

### 解决的问题

解决寒暑假期间孩子作业量大、家长要求不清、老师推进混乱、假期结束作业没完成导致家长不满意的问题。

### 核心结论

作业规划表的价值不只是安排作业，而是建立“孩子承诺、老师推进、家长可见”的共同目标机制。

### 操作步骤

1. 第一天收集学校作业和家长额外布置作业。
2. 区分能在机构完成和需要家庭完成的内容。
3. 由老师协助低年级孩子做规划，高年级孩子可先自主规划。
4. 把总作业量拆分到每天。
5. 每天作业前让孩子确认当天任务。
6. 完成后由老师签字 / 标注。
7. 阶段性反馈给家长。
8. 假期结束后把完整规划和完成记录交给家长。

### 判断标准

- 每个孩子都有独立规划表。
- 家长知道哪些作业机构负责、哪些需要家庭完成。
- 孩子知道每天要完成什么。
- 老师能按表推进，而不是每天临时安排。
- 假期结束能形成可视化结果。

### 常见错误

- 第一天直接写作业，不做规划。
- 老师替孩子承诺，孩子没有参与感。
- 不和家长说明边界，导致后期扯皮。
- 只关注完成，不关注质量和错题。

### AI 调用说明

当用户问以下问题时可引用：

- 寒假作业怎么管理？
- 暑假作业怎么规划？
- 收心班孩子作业没写完怎么办？
- 如何让家长觉得托管专业？

## 9.3 知识卡片：老师只做你检查的事

### 基本信息

- 所属系统：教师管理系统 / 校长经营系统
- 所属年度节点：3月晚辅稳定、10月教师专业度、全年适用
- 所属主题：管理原则
- 适用对象：校长、执行校长
- 置信度：高

### 解决的问题

解决校长反复要求老师执行流程，但老师执行不到位、备课不认真、反馈不稳定的问题。

### 核心结论

老师不会做你要求的事，只会做你检查的事。管理的关键不是“说清楚”，而是“定标准、给工具、做检查”。

### 操作步骤

1. 把要求变成明确标准。
2. 把标准变成表格或清单。
3. 规定检查时间和检查人。
4. 固定频率检查。
5. 对检查结果进行反馈。
6. 对优秀案例做公开表扬。
7. 对长期不达标者进行辅导或淘汰。

### 判断标准

- 老师知道标准是什么。
- 校长知道检查什么。
- 检查结果有记录。
- 老师行为因检查发生改变。
- 流程不再依赖校长临时提醒。

### 常见错误

- 只开会强调，不检查。
- 标准模糊，全靠老师理解。
- 检查没有频率。
- 检查后没有反馈。
- 校长自己下场代替老师干活。

### AI 调用说明

当用户问以下问题时可引用：

- 老师不执行怎么办？
- 老师不备课怎么办？
- 校长怎么管理老师？
- 为什么制度落不下去？
- 如何建立检查机制？

## 11. 质量评价标准

每篇文档进入正式知识库前，建议用以下标准打分。

| 维度 | 评分标准 |
| --- | --- |
| 去噪程度 | 是否去掉直播废话、闲聊、无关互动 |
| 结构清晰度 | 是否从直播时间流转成知识结构 |
| AI 可调用性 | 是否写清适用问题、适用场景、限制条件 |
| 行动化程度 | 是否有 SOP、清单、表格、话术 |
| 保真度 | 是否没有编造原文没有的观点 |
| 培训价值 | 是否能直接变成培训材料 |
| 年度归档准确度 | 是否准确挂到经营节点 |
| 知识卡片颗粒度 | 是否足够独立、可引用 |

### 质量等级

| 等级 | 标准 | 处理方式 |
| --- | --- | --- |
| C 级 | 只是清洗稿，噪音仍多 | 仅存档 |
| B 级 | 有结构，但行动化不足 | 待复核 |
| A 级 | 可培训、可引用 | 进入正式知识库 |
| S 级 | 可变成核心 SOP / 方法论 | 加入核心经营系统 |

## 12. 落地路径建议

## 12.1 第一阶段：搭骨架

目标：先建立系统框架，不追求批量处理。

### 要做的事

1. 建立年度经营行事历。
2. 建立六大经营系统分类。
3. 建立直播原始素材库。
4. 建立结构化文档模板。
5. 建立 AI 知识卡片模板。
6. 选 1 篇直播做样稿。

### 阶段成果

- 一张年度经营地图
- 一个标准文档模板
- 一套知识卡片模板
- 一篇高质量样稿
- 3-5 张示例知识卡片

---

## 12.2 第二阶段：跑通流水线

目标：让 Agent 按固定流程处理一篇直播。

### 要做的事

1. 原始转写稿进入素材库。
2. Agent 基础清洗。
3. Agent 去噪切块。
4. Agent 结构化沉淀。
5. Agent 年度节点归档。
6. Agent 拆知识卡片。
7. 人工复核和评分。

### 阶段成果

- 一条可重复运行的直播处理流水线
- 一套质量评分机制
- 一个可复用 Skill 包

---

## 12.3 第三阶段：批量处理存量直播

目标：把已有直播逐步处理成系统资产。

### 要做的事

1. 按主题优先级处理。
2. 先处理寒假、晚辅、教师管理、招生四类高频内容。
3. 每篇文档拆知识卡片。
4. 同类卡片合并去重。
5. 按年度节点挂载。

### 阶段成果

- 初版年度经营知识库
- 关键节点知识卡片库
- 可用于培训的 SOP / 话术 / 表格

---

## 12.4 第四阶段：年度循环更新

目标：让系统每年越跑越强。

### 要做的事

1. 每年同一节点自动汇总历史资料。
2. 对比今年和去年直播内容差异。
3. 结合实际经营结果复盘。
4. 更新 SOP 和知识卡片。
5. 删除过时内容。
6. 提升 Agent 调用准确度。

### 阶段成果

- 年度经营知识飞轮
- 可复盘、可迭代、可训练团队的业务系统

## 13. 关键风险与边界

### 13.1 不要追求 100% 全自动

知识沉淀可以高度自动化，但不能完全无人审核。

必须人工复核的内容：

- 价格
- 工资
- 合同
- 退费
- 合规
- 安全责任
- 涉及承诺效果的表达
- 本地政策相关内容

### 13.2 不要让长文直接服务 AI

长文适合人读，但 AI 最好调用知识卡片。

否则 Agent 容易被直播残留噪音、长篇案例、无关内容带偏。

### 13.3 不要把讲师经验包装成绝对真理

直播内容很多是经验性表达，沉淀时要保留边界：

- “适用于多数托管机构”
- “需结合本地消费水平”
- “需结合机构定位”
- “需人工确认”
- “不作为法律 / 财务 / 人事专业建议”

### 13.4 不要一开始就整理所有内容

先用几篇高价值直播跑通流程，再扩大。

推荐优先处理：

1. 晚辅效率
2. 寒假托管
3. 寒假招生
4. 教师专业度
5. 团队复盘

## 14. 未来系统运行后的理想状态

当系统跑起来后，你可以这样问 AI：

- 11 月我应该提前准备寒假招生哪些动作？
- 帮我生成一份寒假托管招生方案。
- 老师不备课，我应该怎么设计检查机制？
- 给我一份新老师 7 天培训计划。
- 帮我把寒假直播内容拆成培训课。
- 我下周要开家长会，帮我生成话术。
- 结合年度行事历，告诉我 5 月份该做哪些暑假招生准备。
- 从知识库里找出所有关于“作业规划表”的内容，整理成 SOP。
- 对比寒假托管和暑假托管的流程差异。
- 帮我基于朋阳直播内容生成一份适合我校区的执行清单。

这时，AI 不再只是“回答问题”，而是在调用你的经营知识系统。

## 15. 最终结论

你的需求可以实现，而且非常适合做成一个长期系统。

但关键不是“自动录制 + 自动洗稿”，而是建立一套：

> 全年直播录制 → 转写 → 清洗 → 结构化沉淀 → 年度经营节点归档 → AI 知识卡片拆解 → SOP / 话术 / 培训资产生成 → 经营复盘更新
> 

的知识生产飞轮。

这套系统的核心资产有三件：

1. **年度经营行事历**
    - 决定内容归到哪里。
2. **结构化直播沉淀模板**
    - 决定每篇内容长什么样。
3. **AI 知识卡片库**
    - 决定 Agent 未来能不能稳定引用。

只要这三件事稳定，OpenClaw / Hermes Agent 的输出质量就会明显提高。

这不是一个单次整理项目，而是你托管业务未来的 **经营知识操作系统**。
