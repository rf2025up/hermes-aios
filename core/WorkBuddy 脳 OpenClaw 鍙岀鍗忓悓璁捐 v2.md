# WorkBuddy × OpenClaw 双端协同设计 v2.0

> 2026-04-20 | 基于实际协作场景 + MECE全景 + 原v1.0方案适配
> 核心变化：**弃Hermes，只保留 WorkBuddy（Mac）+ OpenClaw（24h）两端**

---

## 零、先说本质

你的真实痛点不是"双端同步"，而是：

> **你在OpenClaw上聊了1小时、跑了3个自动化任务、产出了2份报告——第二天问我复盘时，我什么都不知道。**

所以设计的核心不是"怎么同步文件"，而是：

> **怎么让OpenClaw产生的高价值内容不丢失，并且能被锋哥在需要时读到。**

---

## 一、双端角色定位（基于实际使用场景）

### WorkBuddy/锋哥（Mac）

**定位：本地高频协作前台**

| 职责 | 具体场景 |
|------|---------|
| 深度对话 | 反思复盘、产品设计、芒格视角分析、教练对话 |
| 本地执行 | 文件操作、Notion写入、docx分析、图片生成 |
| 决策主控 | core/ 宪法层唯一修改权 |
| 记忆主控 | MEMORY.md 唯一修改权 |
| 交班输出 | 每日关键协作→压缩成状态增量→push到共享仓库 |

### OpenClaw（24小时）

**定位：持续运行中枢 + 独立对话节点**

| 职责 | 具体场景 |
|------|---------|
| 定时提醒 | 晨间/午间/运动/晚间4个固定推送 |
| 长任务运行 | 自动调研、批量处理、持续跟踪 |
| 独立对话 | 你在手机/平板上和OpenClaw对话（不在Mac前时） |
| 飞书交互 | 群聊@触发、消息回复 |
| 自动沉淀 | 对话关键节点→日志→共享仓库 |

## 二、原方案没解决的核心问题：OpenClaw对话内容怎么回来

**这是你真正担心的事。**

你会在OpenClaw上做这些：
- 晚上不在Mac前，用手机和OpenClaw聊天反思
- 让OpenClaw跑24小时自动化任务，过程中有对话
- 在飞书群里@OpenClaw讨论业务问题
- 让OpenClaw持续跟进某个家长/某个项目

**这些对话里的高价值内容，目前全部丢失。**

### 解法：OpenClaw对话 → 自动摘要 → 共享仓库

```
OpenClaw对话产生
  ↓ 每次对话结束/每2小时
自动提取关键节点
  ↓ 写入
daily/YYYY-MM-DD-openclaw.md
  ↓ Git push
锋哥下次会话时 pull → 读到
```

### 关键节点提取标准（OpenClaw自动执行）

| 类型 | 示例 | 写入位置 |
|------|------|---------|
| 业务决策 | "明天试课改到下午3点" | daily/ + 更新ACTIVE-TASK |
| 关键沟通 | "斯芮妈妈确认报名" | daily/ + 更新SCOREBOARD |
| 方法论洞察 | "我发现自己又高级逃避了" | inbox/（候选沉淀） |
| 任务产出 | 自动调研报告 | reports/ |
| 情绪/状态 | "今天很焦虑" | daily/（附注） |
| 待办/承诺 | "明天一定打电话给XX" | daily/ + 提醒队列 |

### 摘要格式（OpenClaw每次对话后自动生成）

```markdown
# OpenClaw对话摘要 | YYYY-MM-DD HH:MM

## 对话主题
一句话概括

## 关键决策
- [决策内容]

## 关键承诺
- [谁在什么时间前做什么]

## 候选沉淀
- [可能值得写入MEMORY的内容]

## 情绪信号
- [焦虑/逃避/兴奋/疲惫等]

## 影响的core文件
- ACTIVE-TASK: [是/否，具体变化]
- SCOREBOARD: [是/否，具体变化]
- DECISIONS: [是/否，具体变化]
```

### 3.2 daily/ 目录结构变更

原方案只有一个 `daily/YYYY-MM-DD.md`，现在改为**按来源分文件**：

```
daily/
├── 2026-04-20.md              ← 锋哥写的当天日志（已有）
├── 2026-04-20-openclaw.md     ← OpenClaw当天对话摘要（新增）
├── 2026-04-20-handoff.md      ← 交班摘要（新增）
└── ...
```

**为什么分文件？**
- 避免两端同时写一个文件导致Git冲突
- 锋哥读的时候一眼看出"哪些是OpenClaw发生的"
- OpenClaw读的时候一眼看出"老板白天推进了什么"

### 3.3 同步节奏

| 端 | 动作 | 频率 |
|----|------|------|
| 锋哥 | push到共享仓库 | 每次实质性对话后 |
| OpenClaw | pull共享仓库 | 每小时1次 + 每次对话开始前 |
| OpenClaw | push对话摘要 | 每次对话结束后 + 每2小时（长任务） |
| 锋哥 | pull共享仓库 | 每次会话开始时 |

## 四、交班协议（核心机制）

### 4.1 为什么必须有交班

你白天的推进（锋哥这边）和晚上的推进（OpenClaw那边），如果互不知道，就等于两个脑子各自运行。

**交班 = 让另一端知道"当前状态"**

### 4.2 锋哥→OpenClaw 交班（白天→晚上）

**触发条件**：以下任一
- 你说"我要走了"/"今天到这"
- 22:00自动触发
- 锋哥主动提醒"老板，该交班了"

**产出**：`daily/YYYY-MM-DD-handoff.md`

```markdown
# 交班摘要 | YYYY-MM-DD

## 当前主战役
大航海·阶段二：搭最小获客闭环

## 今天真正推进了什么
- [具体动作1]
- [具体动作2]

## 当前最卡的一点
[一句话]

## 下一步最优动作
[一句话]

## 今晚/明天最需要提醒的事
- [ ] [具体提醒1]
- [ ] [具体提醒2]

## 哪些已写入core
- [文件名]: [变化]

## 哪些只是候选建议
- [内容]

## 禁止误判为"已完成"
- [事项]
```

### 4.3 OpenClaw→锋哥 交班（夜间→白天）

**触发条件**：每天早上7:50（在你8:00坐下之前）

**产出**：推送飞书消息 + 更新 `daily/YYYY-MM-DD-openclaw.md`

```markdown
# OpenClaw夜间运行报告 | YYYY-MM-DD

## 昨晚发生了什么
- [对话摘要1]
- [自动任务产出1]

## 触发的提醒及结果
- 21:00晚间复盘 → [老板回复/未回复]

## 发现的候选沉淀
- [可能值得升级的内容]

## 需要老板确认的
- [ ] [决策1]
- [ ] [承诺1]

## SCOREBOARD是否有变化
[是/否，具体]
```

## 五、OpenClaw对话内容的保护机制

### 5.1 核心问题

OpenClaw的对话上下文**不是持久的**——会话结束、服务重启，上下文就丢了。

所以必须在对话过程中就把高价值内容"抢救"出来。

### 5.2 三层保护

| 层 | 机制 | 何时触发 |
|----|------|---------|
| **实时层** | OpenClaw每次对话后自动生成摘要 | 对话结束时 |
| **周期层** | 每2小时检查长任务，产出中间报告 | 长任务运行中 |
| **兜底层** | 每晚22:00自动检查当天所有对话，补漏 | 每晚 |

### 5.3 长任务运行时的中间产出

OpenClaw跑24小时任务时，不能等到最后才产出。

**规则**：每2小时必须写入一次中间状态到 `reports/`。

```
reports/
├── research-YYYY-MM-DD-HHMM.md    ← 调研任务中间报告
├── tracking-家长名-YYYY-MM-DD.md  ← 跟踪任务报告
└── ...
```

### 5.4 锋哥读取OpenClaw内容的时机

| 场景 | 锋哥动作 |
|------|---------|
| 你问"昨晚OpenClaw有什么进展" | 读 `daily/Yesterday-openclaw.md` |
| 你要复盘某件事 | 读 `daily/` + `reports/` 相关文件 |
| 你做周复盘 | 读本周所有 `daily/*-openclaw.md` |
| 你问"上次在OpenClaw上聊了什么" | 搜索 `daily/` 中关键词 |

## 七、业务变量同步标准

OpenClaw必须实时知道的业务指标（来自SCOREBOARD + 日常协作）：

```
必须同步的业务变量：
├── 学生总数：当前28人 → 目标46人
├── 明天试课安排：斯芮妈妈4/20
├── 转介绍跟进：已开始要家长帮忙
├── 新增家长微信数：→ 目标300
├── 当前阶段：阶段二·搭最小获客闭环
└── 当前主问题：如何把试课→转化→续费跑通
```

这些变量变化时，锋哥必须更新 `core/SCOREBOARD.md` 并push，OpenClaw下次pull就能读到。

## 九、Git规则（双端必守）

| 规则 | 说明 |
|------|------|
| push前先pull --rebase | 避免冲突 |
| 冲突不自动硬解 | 停→报告→等人工 |
| 提交前缀 | `core:` `daily:` `report:` `inbox:` `handoff:` `openclaw:` |
| 防抖 | core每小时≤2次，daily/reports≤4次 |
| 分支 | 只用main，不开分支 |

## 十、最小可运行版本（明天就能跑）

不要一口气做完整系统。**先跑这4件事**：

### 第1步：daily目录分文件（今天就能做）
- 锋哥继续写 `daily/YYYY-MM-DD.md`
- OpenClaw写 `daily/YYYY-MM-DD-openclaw.md`
- 交班写 `daily/YYYY-MM-DD-handoff.md`

### 第2步：锋哥每次实质对话后push（习惯养成）
- 改了core → push
- 写了daily → push
- 不用等"完美"，push了再说

### 第3步：OpenClaw每次对话后生成摘要（配置一次）
- 对话结束 → 自动摘要 → 写入 `daily/YYYY-MM-DD-openclaw.md` → push

### 第4步：交班摘要（最低晚间一版）
- 22:00前，锋哥产出一份交班摘要
- OpenClaw 7:50读取并推送

**做到这4步，双端就已经不是"各聊各的"了。**

## 十一、最容易踩的坑

| # | 坑 | 后果 | 防御 |
|---|-----|------|------|
| 1 | OpenClaw对话不沉淀 | 高价值内容丢失 | **每次对话后自动摘要** |
| 2 | 两端写同一个文件 | Git冲突 | **daily按来源分文件** |
| 3 | OpenClaw直接改core | 系统漂移 | **core写入权铁律** |
| 4 | 交班摘要太长 | 没人看 | **一句话核心+可选展开** |
| 5 | 只push不pull | 信息不对称 | **两端会话开始前先pull** |
| 6 | 把daily当MEMORY | 长期记忆臃肿 | **weekly压缩→archive→MEMORY** |
| 7 | 长任务不写中间状态 | 意外中断全丢 | **每2小时写入reports/** |
