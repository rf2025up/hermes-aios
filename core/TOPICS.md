### 【T11】记忆基础设施升级：MemOS vs 当前方案（GBrain+Markdown）

- **来源**：[锋哥] 2026-04-25，老板发来MemOS链接要求评估
- **核心问题**：当前Markdown+GBrain流程规范的记忆系统，与专业AI记忆OS（MemOS）差距在哪
- **项目**：MemTensor/MemOS v2.0.14（8.7k⭐），Apache 2.0，已有OpenClaw插件
- **评估结论**：

**差距最大的**：
1. **语义检索**——当前只能关键词匹配，不能语义关联（"核心恐惧"搜不到"十几年反复循环"）
2. **跨Agent记忆共享**——锋哥/Star各记各的，靠git同步有延迟有冲突风险

**当前方案优势**：
1. 人类可读（Markdown直接打开）
2. 零依赖（不需要Redis/Neo4j/Qdrant）
3. 业务逻辑深度绑定（三问拦截、能力链7环全在MEMORY里）
4. 可控性100%（每条记忆能看到改到删到）

**GBrain方法论 = 用流程规范弥补架构缺陷**：
| GBrain做了什么 | MemOS对应能力 |
|---|---|
| 强制读取前缀 → 解决"不读记忆" | 内置API调用，天然读取 |
| 变更日志 → 解决"改了什么不知道" | 图结构变更可追溯 |
| 梦境周期cron → 解决"记忆整理" | MemScheduler自动调度 |
| 幂等同步 → 解决"重复写入" | API天然幂等 |
| 30天蒸馏 → 解决"记忆膨胀" | 生命周期管理 |

- **行动建议**：先记下来不急着装。等产品内核收完、记忆量上一个台阶时再升级
- **安装成本**：2分钟（OpenClaw插件已有），但迁移+调试至少半天
- **当前状态**：🟡评估完成，待时机
