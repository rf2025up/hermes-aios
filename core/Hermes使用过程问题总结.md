## P1: 记忆丢失（系统性问题，反复发生）

### 问题概述

自2026-05-04迁移到WSL2以来，"不记得之前说过/做过的事"反复出现，严重影响协作连续性。

### 已发生的具体案例

| # | 日期 | 场景 | 丢了什么 | 后果 |
|---|------|------|----------|------|
| 1 | 05-04 | WSL2迁移后首次对话 | 知识洗稿工作流、整理核心文档任务、daily日志路径 | 搜了5次session_search全部无结果，用户反复纠正 |
| 2 | 05-04 | cron梦境周期 | memory工具不可用 | 只能降级为直接patch文件 |
| 3 | 05-05 | 全天对话 | aios/core目录未迁移 | 所有文档找不到，相当于"工作手册丢了" |
| 4 | 05-05 | 建文档规则 | 锋哥要求建立"问题总结文档" | 未记住，未执行 |
| 5 | 05-06 | 音频转写任务 | 之前确认用DashScope API | 装了本地FunASR硬跑，浪费7.5小时 |
| 6 | 05-06 | 上下文压缩 | 152条消息被删，摘要生成失败 | API转写决策丢失，导致案例5 |

### 根因分析

**记忆丢失不是偶发bug，是架构缺陷。有三个层面：**

**层面1：上下文压缩不可控**
- Hermes在对话过长时自动压缩，删除旧消息
- 压缩时如果摘要生成失败（"summary generation was unavailable"），信息永久丢失
- 没有hook让Agent在压缩前主动捕获关键信息

**层面2：持久化记忆太弱**
- memory只有2200字符限制，存不了几条重要信息
- 被迫做取舍，丢掉"看起来不那么关键"的内容
- 存了也可能在新会话中不被检索到

**层面3：跨会话检索靠不住**
- session_search依赖摘要质量，摘要不可用就等于失忆
- 没有语义检索能力，"问题总结文档"搜不到"错误记录"
- 知道去搜 ≠ 能搜到

**死循环：记忆差 → 写规则 → 规则也要靠记忆 → 又忘了规则**

### 当前缓解措施（不够）

1. SOUL.md写了"执行前先读项目文档" — 依赖我"想起来这条规则"
2. memory里存了关键事实 — 2200字符快满了，存不下更多
3. INDEX.md做了文档索引 — 但建立文档的规则本身也需要记忆
4. session_search跨会话搜索 — 摘要质量不可控

### 根治方向

**启用MemTensor插件（memos-local-plugin）**，已安装在`~/.hermes/plugins/memos-local-plugin/`，但从未初始化运行。

| 记忆问题 | MemTensor怎么解决 |
|----------|------------------|
| 压缩前丢信息 | `on_pre_compress` hook自动捕获关键信息 |
| 跨会话失忆 | L1/L2/L3分层记忆，SQLite持久化 |
| 搜不到 | 语义检索（embedding），不依赖关键词 |
| memory字符限制 | 无限制，按需检索注入 |
| 不知道之前怎么定的 | `on_turn_start`自动检索相关记忆 |

**状态：待执行。需要锋哥确认后初始化。**

## P2: 文档管理混乱（重复创建、无统一索引规则）

### 问题概述

core/目录有49个md文件，其中至少5个是乱码重复文件。文档之间缺乏统一的建设规则，导致：
- 重复创建（同一主题多个文档）
- 不知道该往哪个文档写
- 文档之间关系不清

### 根因

没有"建文档的规则"。每次需要记录东西时，凭感觉建文档，没有先查索引、没有遵循统一分类。

### 已执行的纠正措施

1. 清理了乱码重复文件（4个）
2. 建立本文档（Hermes使用过程问题总结）
3. 制定建文档总纲（见下方"建文档总纲"章节）

## 建文档总纲

### 总则

> **建文档前必须先查INDEX.md。不确定放哪就问，不要凭感觉新建。**

### 文档分类与职责

**A类：必读文档（每次对话加载，核心身份）**

| 文档 | 存什么 | 不存什么 |
|------|--------|----------|
| SOUL.md | 行为准则、铁律、人格定义 | 具体操作步骤、业务数据 |
| USER.md | 锋哥偏好、习惯、角色 | 一次性信息、临时任务 |
| MEMORY.md | 长期稳定事实（环境、配置、结论） | 任务进度、每日日志、时间敏感信息 |

### 建文档流程

```
需要记录信息
    ↓
查INDEX.md → 已有对应文档？
    ├─ 有 → 直接追加/更新该文档
    └─ 没有 → 问自己：
              ├─ 属于A/B/C/D哪类？
              ├─ 真的需要新文档，还是可以放入已有文档？
              ├─ 如果确实需要新建 → 建完后立即更新INDEX.md
              └─ 如果不确定 → 问锋哥
```

### 禁止事项

1. **禁止不经INDEX.md查询直接新建文档**
2. **禁止创建与已有文档功能重叠的文档**（先用搜索确认没有同类文档）
3. **禁止用文档替代action**（建文档本身不等于解决问题）
4. **禁止在core/根目录堆积临时文件**（临时文件放inbox/）
